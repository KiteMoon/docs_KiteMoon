    Dockerfile文件编写

    简介

    镜像构建其实就是一个叠叠乐的过程,通过执行一系列的命令来附加不同的层,最后将层合并为镜像来实现

    创建流程

    创建镜像的过程也就是我们编辑dockerfile的过程,具体过程如下

    注意,文件名必须要是Dockerfile

    导入基础镜像-FROM

    首先第一步,我们需要导入一个基础镜像,基础镜像提供一个最底层的OS(这个OS不完整,内核啥都是用的宿主机),这里我们选用centos,作为基础镜像.

    FROM一个文件中只能出现一次,只能在最开头,除非要多阶段构建

    FROM:继承基础镜像

    FROM centos:7 # 指定了centos7惊喜那个

    构建中间层-RUN

    我们导入基础镜像后,需要对镜像进行定制化,例如导入运行环境啊,引入代码啊,准备缓存目录什么的

    这些步骤就是中间层要做的,中间层的代码是RUN,一个容器可以有执行多次RUN,每执行一次RUN代表新生成一个层

    因此不建议部署太多的RUN,额外的层会带来额外的开销

    RUN后面跟应该跟随的是一个Linux命令,具体如下

    FROM centos:7
RUN useradd kitemoon
RUN go build ./

    上面的例子其实不好,因此一条命令就占用了一个层,会导致不必要的层出现,因此我们最好这样写

    FROM centos:7
RUN useradd kitemoon && go build ./

    导入文件-ADD&COPY

    导入文件分为两种,COPY和ADD

    一般来说,我们不会在容器中去下载过多的软件,这一系列软件也包含了curl和wget,但是没了他们我们业务代码就不好导入了,此时就可以使用ADD指令,将我们本地文件复制到容器之中

    你会发现很多开源项目很喜欢把dockerfile放到根目录,因为这样很好做CICD

    具体的实现指令如下

    ADD命令如下

    注意,ADD如果移动的是压缩包,会自动解压

    ADD <宿主机文件路径> <容器内路径>
ADD ./install.tar /mnt/app/

    copy命令如下

    COPY是不会自动解压的,因此你还得开个RUN,然后去解压

    copy <宿主机文件路径> <容器内路径>
copy ./install.tar /mnt/app/

    请注意,COPY和ADD都是dockerfile的文件,他不是系统命令,因此是不支持你添加-r参数来实现连着根路径一起拷贝过去,他默认会将文件夹下的内容拷贝过去,因此你得指定路径实现曲线救国

    # 把temp文件夹下的内容拷贝到root
COPY temp /root/
# 查看root会发现里面是 1 2 3
# 把temp文件夹下的内容拷贝到/root/temp下
COPY temp /root/temp
# 查看root会发现里面是 temp

    引入环境变量-ENV

    在容器中,有个很重要的概念就是环境变量,因为容器大概率跑的是无状态的应用,但是你的程序极大概率是要引入配置文件的(比如外部数据库的账号密码等),这种情况下,就需要使用环境变量,在环境变量中存储账号密码信息,你的程序连接数据库的位置采用环境变量,这样有效的避免了无意义的卷映射.

    没必要为了改一个配置文件去设置-v参数,特别是多容器环境下,更不应该如此,也很难实现,这就是为啥有configmap的存在

    引入的代码如下

    # 格式1
ENV <key> <VALUE>
# 格式2
ENV <key>=<value> <key>=<value>
ENV mysqlHost 192.168.92.100

    支持用-e <key>=<value>去覆盖这个环境变量,CLI的优先级高于dockerfile的.

    之后你去容器里面拉环境变量就可以看到我们设置的环境变量了

    灵活运用可以让体验优化,比如默认build的时候就携带了ENV,采用本地SQL,用户可以通过-e让他修改为mysql

    配置工作目录-WORKDIR

    配置工作目录比较重要,他关系到你RUN和CMD执行的位置

    RUN和CMD执行的默认位置为/,但是你指定了工作目录后,会以工作目录来执行,也就无需提前开个RUN进入到指定路径中.直接就可以在工作目录下工作

    WORKDIR <工作路径>
WORKDIR /mnt/app

    修改workdir后,你bash进去的路径也会变成你设置的工作路径

    暴露的端口号-EXPOSE

    在WEB服务上,我们希望能够暴露端口出来,提供给我们的其他POD或者用户使用,就需要理由该特性

    你不设置...其实也可以访问

    支持多条EXPOSE语句

    EXPOSE <端口号>
EXPOSE 80

    容器使用的用户-USER

    容器大多数情况下都会自动使用root,但是有的软件比较挑剔,比如是从裸机迁移到容器中,就会强制要求用户使用非ROOT用户,此时就需要使用USER参数去实现了

    minikube好像就必须运行在非root用户上

    USER <用户名>
USER KiteMoon

    编译入参--ARG

    ARG可以用于设置编译镜像时传入的参数,来实现基于模板构建

    有的用户很挑剔啊,LABEL必须要是自己公司的名字,但是我们又不可能手动改,就用这个

    用得少,我也没用过,这里就不多写,怕误导

    最后的步骤-CMD

    之前容器的状态部分我说过,容器停止的依据不是基于我们exec进去的shell,而是基于容器CMD所执行语句的状态决定的

    在容器启动时,会执行最后一个CMD语句,在该语句执行结束后,或者发生错误退出后,容器会自动进入停止状态,这也是我们希望看见的.

    因此,请将你程序的启动指令放在这里,作为启动命令,如果你的程序执行完毕,那么容器也会停止,资源也会自动释放

    如果你在docker run的过程中后面加了命令,则会替换掉你CMD的命令,你容器的启动与停止也决定于你执行的这条命令

    CMD <启动程序的代码>
CMD /proess

    入口点-ENTRYPOINT

    ENTRYPOINT可以和CMD共存,请注意

    具体实现的功能如下

    功能1-透传参数

    有的时候,我们希望用户启动容器的时候,为我们导入一个参数信息,例如容器内程序的启动密码等,但是直接使用环境变量会提高软件的使用难度,因此就可以为他添加入口点

    入口点支持将用户的的在启动容器时输入的参数透传到容器内部

    语法如下

    ENTRYPOINT <启动命令>
ENTRYPOINT /app/run
ENTRYPOINT [ "/app/run" ]

    entrypoint 其实并不是直接屏蔽了用户命令后面跟着的参数,而是直接当做入参,因此你可以把他当做启动密码的输入点.

    # 当我执行容器的时候,就实现了参数传递
docker run sss:v1 123456
# 执行上述参数的时候,约等于在容器中执行了
/app/run 123456

    我在网上找了个好的例子,myip

    myip是ipip提供的一个接口,能看到你自己的IP地址

    容器内部是使用curl去请求一个URL,然后返回值

    FROM centos:7
RUN yun update && yum install curl
ENTRYPOINT [ "curl" "-s" "http://myip.ipip.net" ]

    正常情况下执行会返回

    docker run ipip
当前 IP：xx.xx.xx.xx 来自：xx市 xx

    如果这个时候我们想查看http头,则无需修改dockerfile或者进入容器,只需要这样执行指令即可

    docker run ipip -i
<header信息>

当前 IP：xx.xx.xx.xx 来自：xx市 xx

    在这里,ENTRYPOINT就实现了参数透传的功能,将用户补充的-i 透传到内部中,此时内部执行的代码就变成了

    curl -s http://myip.ipip.net -i # <-这就是刚才透传进来的

    功能2-作为初始化脚本

    ENTRYPOINT其实是可以作为一个前置步骤放到CMD前面的,可以降低对环境变量的依赖

    例如我的容器需要先判断启动模式,是客户端模式还是服务器,用户在启动的时候会在启动命令中输入server或者client,之后该参数将会透传到容器内部,被ENTRYPOINT捕获,并通过该参数的值去判断是执行服务端初始化还是客户端初始化

    FROM centos:7
ENTRYPOINT ["/init/init.sh"]
CMD /app/run

    在上述的dockerfile文件中,我们用户如果输入的是server,则入口点会自动执行/init/init.sh server进行服务器初始化 最后启动主程序,这就是一种使用方法

    如果对这个感兴趣可以去看下redis的dockerfile,在他的启动命令中,有一个参数要求用户是服务端启动还是客户端启动,也是利用了入口点的特性.

    ARG-dockerfile动态传参

    一般是用在CICD上多一些

    ARG关键字用于在dockerfile内放置一个变量,并接受用户在build时所附加的参数上,并且支持默认参数(即为你没有传递该参数时默认使用的值),这样可以有效降低dockerfile的编写成本呢

    ARG只在构建过程中有效,构建后就没了

    以后改版本号啊什么的都可以用ARG,然后Jenkins在编译时提供版本号参数,实现一个dockerfile适应多个版本

    关键字如下

    # 这里是dockerfile
FROM centos:7
ARG USERNAME # 不提供默认值变量
ARG SHELLPATH="/bin/bash" # 提供默认值
RUN useradd $USERNAME -s $SHELLPATH # 调用变量创建这个用户,并指定shell
USER $USERNAME
CMD echo "HELLOWORLD"

    警告,请不要在dockerfile外使用ARG提供的变量信息,这是无效的!!!!特别是在CMD中去调用这些变量返回的只会为空,如果你有需要,请用ENV去实现.

    之后我们只需要在构建过程中添加选项--build-arg,并在后续使用key=value的格式去赋予值即可

    docker build --build-arg USERNAME=kitemoon .

    dockerfile大小优化

    我们在之前的例子中,都是from的centos,但是实际上centos的包很大,也包含了很多我们用不上的功能,因此容器的镜像大小很大,这很不利于我们在生产集群中的部署(分发个包下去,十分钟内整个控制平面全是image),因此就有了这个部分,对镜像大小进行优化

    生产环境中不建议使用系统级别的镜像,很大很多,也不安全

    优化大小一般是两个部分:底包大小优化和层优化

    底包优化

    底包优化的话,我们一般使用底包是alpine镜像,centos的低包太大了

    centos7的底包是204MB alpine是

    因此,我们更多的使用alpine去构建

    alpine的软件包管理器是apk 添加软件是adduser命令(-D为不要密码)

    因此我们可以修改一下我们的程序

    FROM alpine:3.12
RUN adduser -D kitemoon
CMD echo "helloworld"

    构建速度回很快,而且很小

    Step 1/3 : FROM alpine:3.12
 ---> 24c8ece58a1a
Step 2/3 : RUN adduser -D kitemoon
 ---> Running in 4929d5560df0
Removing intermediate container 4929d5560df0
 ---> 2616f3daf04e
Step 3/3 : CMD echo "helloworld"
 ---> Running in be25277c5557
Removing intermediate container be25277c5557
 ---> dc2daf7f715d
Successfully built dc2daf7f715d

    大小也很小

    而且功能依旧正常,因此,选择合适的底层镜像是非常有必要的

    注意,对于java等还价下,建议去官网直接找他们的基础镜像,有特殊优化

    层优化

    对于层优化,我们可以先看下我们的镜像有多少层

    关键命令是docker history <image-name>,这里我随便拉出来一个比较标准的层结构,即可从中分析每个层占用他了多少空间

    一般来说,我们对层的优化,都是让他尽可能的变少,哪怕你看着他占用空间几乎没有,但是他依旧是额外的开销,因此我们很喜欢在一个RUN中执行完所有要初始化的步骤,下面是对比图

    正常情况下,我们使用多个RUN去执行操作,可以看到,我们有七层(不计算FROM层)

    如果直接一个RUN + 连接符实现一次完成操作,那么层数就会变少,6层就搞定了

    分阶段构建

    挺多时候,我们的代码拉下来是要先构建一次,然后生成二进制文件,最后运行这个二进制

    这就产生了一个问题,在编译的过程中,会产生太多的依赖文件了(比如golang下要给你下一堆依赖包用于构建),但是这些依赖文件你是不需要的,你只需要二进制文件,因此我们需要分为两个阶段去构建,阶段一构建,阶段二去执行二进制

    为了避免这个情况,docker设计了分阶段构建机制,你可以通过二次FROM的方式,让docker只使用第二阶段进行构建,同时不丢失第一阶段的构建结果

    简单举个例子,你前四部是阶段1,在第四步完成了构建,然后第五到第八步才是第二阶段,docker支持运行第1-4步,并且让你在第五步把前面生产的产物给取出来,随后丢弃到第1阶段的内容,只构建后续的步骤

    其实很多时候,外部CICD已经给你生成好二进制了,这里倒也不需要去分阶段构建了

    我们这里需要使用的关键字是FROM scratch

    具体命令如下

    FROM golang:1.20 AS gobuild # 这里不能少,标识这里是构建阶段gobuild
WORKDIR /opt
COPY hello.go /opt
RUN go build /opt/hello.go # 编译出产物
# 构建步骤2
FROM scratch 
COPY --from=gobuild /opt/hello . # --from代表从哪个编译阶段中提取文件
CMD "./hello" # 执行

    dockerfile的一些常见构建错误

    Error response from daemon: dockerfile parse error line 4: ARG names can not be blank

    该错误是因为你用ARG的时候,在赋予变量默认值的过程中打空格了

    # 错误例子
ARG SHELLPATH = "/bin/bash"

    请不要打空格

    # 正确操作
ARG SHELLPATH="/bin/bash"
