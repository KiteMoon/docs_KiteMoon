    什么是容器

    容器你可以理解为是一个个的集装箱,集装箱下面是货轮,货轮就是我们的物理机,集装箱就是容器

    容器的优势就是小和轻便,同时支持预制环境,我们可以在集装箱中准备好基础的环境依赖,我们只需要把业务代码放上去即可,有效减少配环境的时间

    这里说的基础环境依赖就是基础镜像

    我们通过基础镜像可以进行二次打包,将我们的业务代码放到基础环境上,再统一打包(基础镜像+业务代码),就是我们的业务镜像

    基础镜像和业务镜像的区分其实不是太明显,基础镜像可以是centos,那么我给他安装了java后他就变成了业务镜像,但是我们再将我们的业务代码放到安装java的centos镜像上的时候,这个业务镜像又变成了基础镜像,所以不要太过于死板

    我们将业务镜像在我们的node上运行起来,这就是容器(container)

    容器的架构图

    下面这张图是一个对比图,对比了传统VM架构和Container架构

    • Infrastructure
      • 物理平台,一般是服务器
    • Hypervisor
      • 虚拟化软件
      • ESXI,PVE,FusionCompute
      • 也支持你用HostOS后安装KVM等软件实现该层
    • Guest OS
      • 虚拟机的操作系统
    • Bins
      • 依赖

    • APP

      • 应用

    • HostOS

      • 宿主机系统
      • Centos等
    • container Manager
      • 容器管理平台
      • docker/containerd
    • container
      • 容器

    可以从上面看出来,VM架构下,针对不同的APP部署了不同的VM,每个VM下都有独立的Guest OS,因此隔离性较强,缺点则是每个APP都要开销一个GuestOS性能浪费严重,并且由于资源利用率的原因,大多数公司会部署多个业务在一台主机容易产生版本冲突

    Container架构下,我们在Host OS上安装了容器管理器后启动容器,容器共享HostOS的内核和资源,资源利用率高,并且由于无需重新拉起GuestOS,直接调用本机资源,因此启动速度快(秒级),但是由于共享底层HostOS,因此安全性也会更低

    什么是docker

    Docker不行啦,哈哈

    Docker是一个开源的软件项目,他实现了container标准,是一个轻量化的容器虚拟化平台

    并且支持网络设计等

    关键架构

    • Docker Client
      • Docker的客户端,用于执行Docker相关命令
      • docker pull这类命令都是用Client实现
    • Docker Daemon
      • Docker守护进程,用于实现对Docker的监听和管理
      • 可以理解为Docker的Server
      • Redhat开源的Podman采用对单个容器守护进程的解决方案,Docker实现一个程序,用一个程序守护容器的方式,前者单点故障概率会更低一些
    • Docker Image
      • Docker镜像,可以看做一个模板,基础环境被封装到其中
      • 可以把业务环境封装到其中后运行
      • 只读镜像,可用于启动容器
      • 容器删除后镜像恢复原状
    • Docker Container
      • Docker的容器,可以看做把Image给跑起来
      • 产生的数据位于可写层,不会污染镜像,因此可以多个容器复用一个Imag

    OCI

    Open Container Initiative 开放容器标准

    这是一个通用标准,只要实现该标准,都能被K8s纳管

    Containerd

    行业标准的容器运行时，强调简单性、稳健性和可移植性

    Docker为了兼容OCI标准,将Runtime和管理功能从Docker守护进程中分离出来,支持在不启动Docker的情况下也能直接通过Containerd

    目前containerd已经完全独立出来了,和Docker关联也不是很大了

    containerd主要是轻便,docker作为一个商业应用,他实现了很多很多功能,比如编排啥的,但是我们用K8s的时候是不需要这些的,这些由K8s去实现,因此K8s最好是containerd

    Runc

    Runc是属于Containerd的一部分,负责容器启停,状态管理等服务

    简单来看,Runc就是Containerd的钥匙,通过RunC去控制containerd

    容器运行时标准,是Docker贡献给云原生基金会的.

    他实现了一套对容器运行状态的管理标准,并添加了一系列的状态时,也提供了一套容器启停,资源隔离等功能.

    有Runc后,无需通过docker引擎,即可直接控制containerd

    关于Runc和Containerd的关系(他们都是容器进行时)

    是的，containerd也是一个容器运行时。containerd是一个用于管理容器生命周期的工具，它负责容器的创建、运行、停止和销毁等操作。与runc不同的是，containerd提供了更丰富的功能和更高层次的抽象，使得它可以用于构建更复杂的容器管理平台。containerd通常被用作容器编排系统（如Kubernetes）的底层组件，而runc则是containerd的一部分，用于实际运行容器。因此，虽然它们都是容器运行时，但它们的定位和功能略有不同。

    cgroup

    cgroups，是Linux内核的一个功能，用来限制、控制与分离一个进程组的资源。

    cgroup支持对一个或者一组进程进行资源限制,并且支持被容器进行定额分配,例如50%的CPU资源这种操作

    在这里,我们不必太过于去了解这个,这是一条全新的技术路线,他也可以实现容器的效果,他通过向容器分配指定的CPU和内存,还有硬盘存储空间来实现隔离,因此,如果在你不进行设置之前,docker会默认使用cgroup功能来实现容器隔离
