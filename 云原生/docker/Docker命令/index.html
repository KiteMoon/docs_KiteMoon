    下载镜像

    从容器仓库中下载镜像

    注意,你不佳前缀默认是去默认仓库(docker.io)去拉,如果你需要从自己的仓库拉取,请使用下面的方式

    注意下面的命令,xxx是你仓库的域名,zzz是你项目名称(harbor)或docker的用户名,tttt是镜像的名称,yy是仓库的tag,tag可以简单理解为镜像版本

    # 直接从仓库拉
docker image pull nginx
# 从私有镜像拉取
docker image pull registry.xxx.xxx/zzzz/tttt:yy

    返回如下

    修改镜像名称(打TAG)

    修改不会占用额外的空间,而是会直接改名,你看到两个一样大小的,其实用的是一个文件

    很多时候你拿下来一个基础镜像,二次加工后要push到自己的仓库,就需要该命令

    如果修改原有的镜像名称存在TAg,请不要忘记TAG,或者直接image id

    docker tag <原来镜像的名称/ID> <新的镜像名称>
docker tag nginx:latest blog.13x.cc/nginx/nginx:v20

    该命令无返回值,你查看镜像列表即可看到效果

    REPOSITORY                                                                      TAG         IMAGE ID       CREATED         SIZE
blog.13x.cc/nginx/nginx                                                         v20         a8758716bb6a   3 months ago    187MB
nginx                                                                           latest      a8758716bb6a   3 months ago    187MB

    删除容器

    删除要分为两个步骤,停止 删除.你不能直接删除一个在运行中的容器,这里假定你已经删除了容器

    docker remove <容器ID>
root@VM-20-5-ubuntu:~# docker remove 7625d02032d8

    该操作会返回容器的ID,代表已经删除

    进入容器

    进入容器前,需要确保你容器中有个shell,否则会因为没有shell被弹出来

    进入容器很简单,主要一开-it也就是生成tty和交互式终端

    docker exec -it <contianerd-id> <shell位置>
root@VM-20-5-ubuntu:~# docker exec -it 7428381f89ed /bin/bash

    进入中断后会发现名称啥的都变了

    root@7428381f89ed:/# ls

    退出可以用exit

    注意,你这里的退出不会导致容器停止,容器停止是基于dockerfile中CMD执行那条命令决定的,那条语句结束了,容器停止了

    特殊的启动容器方法

    该方法只适合于调试

    当我们不希望使用image自带的CMD命令,而希望提供给我们个交互终端,我们自己进去调试运行,就可以使用下面这种命令方式

    这种方式下,我们一旦退出容器,容器就会自动停止

    这种方式下,容器原有的CMD命令将不会执行

    docker run -it <image-name> <shell-path>
docker run -it nginx /bin/bash

    查看日志

    查看日志和k8s一样,就不过多赘述了

    -f代表持续输出

    有的容器运行久了请用tail处理下,别卡死了

    docker logs -f <container-id>
docker logs b116a -f - tail 1

    返回如下

    {"time":"2024-01-30T08:11:09Z","latency":"100.493µs","method":"GET","uri":"/","status":200,"error":""}

    映射端口

    映射端口较为简单,-p即可,别忘了顺序,先宿主机再容器内

    一个-p只能映射一个,要多个请多次-p

    docker run -p <宿主机端口>:<容器内端口> <image-name>

    如果需要查看映射的端口,请使用下面的命令

    netstat -ltup | grep <映射的端口>

    效果如下

    root@VM-20-5-ubuntu:~# netstat -ltup | grep 8008
tcp        0      0 0.0.0.0:8008            0.0.0.0:*               LISTEN      1879042/docker-prox 
tcp6       0      0 [::]:8008               [::]:*                  LISTEN      1879049/docker-prox

    容器内一些网络细节

    在docker中,已经为你准备好网卡了,正常情况下来说,你docker内有多少个网络,你的宿主机就会有多少张虚拟网卡,你可以直接通过容器的IP地址+端口直接访问到容器,而无需使用端口映射

    两个容器网络之间还是会隔离的,但是对于宿主机来说,默认接入了所有的网络

    如果我们希望尝试从宿主机访问容器内端口,可以尝试用下面的命令提取IP

    docker inspect <containerd-id> | grep IPAddress

    应该会返回这个

    root@VM-20-5-ubuntu:~# docker inspect 742 | grep IPAddress
            "SecondaryIPAddresses": null,
            "IPAddress": "172.17.0.5",
                    "IPAddress": "172.17.0.5",

    上面命令就是搜索的,你可以不用管道符自己翻

    然后就可以用该IP地址去访问了

    oot@VM-20-5-ubuntu:~# curl 172.17.0.5
<!DOCTYPE html>

</head>

    数据持久化

    这里是Docker版本的,K8s的后面去自己看

    正常情况下,容器分为读写层和镜像层,镜像层是只读的,我们所有对容器的文件操作都是发生在读写层.

    容器每次被构建,都会以最干净的状态去启动

    但是读写层存在一个问题,读写层生命周期和容器同步,也就是说,容器存在时,读写层存在,容器删除后,读写层也被删除了

    该特性就不太利于持续部署,特别是集群环境下,而且如果容器挂了里面的数据也就丢失了,而且也不太方便读取配置文件这种操作(生产环境和开发环境配置不同,你配置文件封装到了镜像层,你没法去持久化修改)

    典型例子就是mysql容器.你要是没注意重建了,数据就消失了

    因此我们需要使用绑定卷的方式,操作很简单-v即可

    绑定卷:将虚拟机的某一路径和物理机的某一路径绑定,因为我们是将宿主机绑定到存储卷上,因此叫绑定卷

    路径绑定请使用绝对路径,映射路径无需添加尾部/

    docker run -v <宿主机路径>:<容器内路径> <image-name> 
docker run -v /mnt/nginx/html:/usr/share/nginx/html

    拷贝文件到容器

    用的真的很少

    有的时候你希望把你本地的一个文件换到容器内种,就可以用到这个命令

    docker cp <本地文件路径> <容器ID>:<容器内路径>
