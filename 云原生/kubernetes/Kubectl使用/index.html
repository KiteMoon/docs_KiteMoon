    Kubectl使用

    常用的kubectl操作和故障排除

    BASH补全

    命令基于你使用的bash,其他的自行文档

    yum install -y bash-completion
source <(kubectl completion bash)
echo "source <(kubectl completion bash)" >> ~/.bashrc
source /usr/share/bash-completion/bash_completion
source <(kubectl completion bash)

    配置文件的选择

    配置文件选择基于环境变量和家目录下的配置文件中选择

    环境变量:KUBECONFIG

    配置文件:`~/.kube/config

    集群切换

    正常k8s的config是单个集群,但是也支持多集群,切换命令如下

    考试中也有这个哦,第一题都让你切换

    kubectl config use-context <集群名>

    增加命令

    对集群进行增加命令有两种

    create

    create用于新创建环境,当我们执行create的时候.如果发现集群中的资源已经存在,则会返回错误,因此只适合于新创建的环境

    拿这玩意去做job类创建挺不错的,避免错误多次创建同一种任务

    kubectl create -f <文件>

    apply

    apply适合于修改资源,他会将所有的操作重新应用一遍,可以理解为刷新配置,如果有变化,则更新,如果没有变化则无事发生

    kubectl apply -f <文件>

    直接创建POD

    只建议测试时使用

    可以通过create命令直接拉起一个POD,但是只支持单个容器

    kubectl creaate deployment <部署对象名> --image=<容器镜像地址> -n <命名空间>

    获取某个POD的部署文件

    K8s中的资源都有属于自己的描述文件,即为如何创建自己,如何实现自己,可以利用下面的命令获取

    kubectl get deploy <部署名> -oyaml

    -o是输出 yaml是用yaml格式输出

    如果你忘记一个deployment的结构的话,可以使用下面的命令输出

    kubectl get deploy <部署名> -oyaml > ./create.yaml

    同时我们也可以在创建过程中直接获取yaml文件,而不是等创建完成后再提取

    kubectl create deployment <部署名> --image=<镜像名> --dry-run=client -oyaml

    --dry-run=client 代表干运行,即为不执行,只是模拟,此时在利用导出参数既可获得yaml

    同时应用多个文件

    你希望一条命令执行几个yaml文件的时候可以使用

    使用该语法后,路径是不能补全的,不是很推荐

    kubectl apply -f <文件1>,<文件2>

    删除命令

    删除命令较为简单,你带上Delete即可,需要注意,同时支持按照NAME删除和通过yaml文件删除,如果你需要删除一个大的部署方案,建议用yaml操作,避免漏掉某些配置

    # 直接删除
kubectl delete <资源类型(pod/deployment/service...)> <name>
# 基于yaml 
kubectl delete -f <filePath>

    范例

    root@ansible ~]# kubectl delete deployment nginx -n default
deployment.apps "nginx" deleted

    基于deployment去部署的POD,你即使直接删除了pod,deployment也会直接去新创建一个POD,因此你得去删除deployment

    注意:删除这种操作是要指定命名空间的

    查询命令

    关键字是get

    关键参数有 -a:列出所有的命名空间

    -o wide:显示详细信息

    -n 显示指定的命名空间

    kubectl get <资源类型> <资源名> -n <命名空间>

    范例如下

    [root@ansible ~]# kubectl get deployment coredns -n kube-system
NAME      READY   UP-TO-DATE   AVAILABLE   AGE
coredns   2/2     2            2           2d20h

    注意 node和clusterrole无命名空间隔离性,全都可用,无需指定

    查看是否有命名空间

    刚才提到node和clusterole是无命名空间的,那么哪些是有的呢?请使用下面的命令查询

    记忆一下常用的即可

    # 查看有命名空间的
kubectl api-resources --namespaced=true
# 查看无命名空间的
kubectl api-resources --namespaced=false

    使用排序

    很多时候,我们希望系统按照一定顺序返回值,比如按照名称排序,我们可以利用--sort-by函数去筛选

    筛选过程中采用.进行选中,类似于go的字典索引

    这个需要使用索引语法,一步一步的去索引(yaml)

    可以配合kubectl get <资源类型> <对象> -n <命名空间> -o yaml的方式去提取出来索引信息,然后在下面用.去索引出来

    kubectl get <service type> --sort-by=<依据字段>

# 实例

[root@ansible ~]# kubectl get svc -n kube-system --sort-by=.metadata.creationTimestamp

    查看与筛选POD和NODE的标签

    之前说过脏节点的概念,脏节点会为自己打上标签(Label),我们如果想要查看这些标签,可以使用扩展符--show-labels

    kubectl get <资源类型> <对象名> -n <命名空间> --show-label
# 实例
[root@ansible ~]# kubectl get node --show-labels

    同时,如果我需要使用筛选功能,比如筛选所有的控制节点,同样可以使用-l参数实现,并且支持多种条件语法

    -l无需使用索引语法,他只会去查找label中的标签

    # 使用不等于语法
[root@ansible ~]# kubectl get node -l beta.kubernets.io/os!=windows
NAME              STATUS   ROLES           AGE     VERSION
master-1.13x.cc   Ready    control-plane   2d23h   v1.28.1
master-2.13x.cc   Ready    control-plane   2d23h   v1.28.1
master-3.13x.cc   Ready    control-plane   2d23h   v1.28.1
worker-1.13x.cc   Ready    <none>          2d23h   v1.28.1
worker-2.13x.cc   Ready    <none>          2d22h   v1.28.1
# 使用等于语法
[root@ansible ~]# kubectl get node -l beta.kubernets.io/os=windows
No resources found
# 使用直接筛选
[root@ansible ~]# kubectl get node -l node-role.kubernetes.io/control-plane=

    更新资源

    更新资源目前有四种方式

    edit-文本编辑器修改现有资源

    有的时候我想直接修改现有资源用于调试,或者因为复杂度的原因,不方便直接使用yaml去更新文件,就推荐使用edit

    比如之前提到的kubeadm初始化文件,那个文件一执行就容易炸,还会导致全部节点被刷新,但是我们只想要修改一点点,就可以使用edit

    kubectl edit <资源名称> <对象名> -n <命名空间>

    set-直接修改现有资源

    set的功能和edit是一样的,不同的是,set使用了命令行修改,这让他可以被插入到shell脚本中去,或者日常修改资源使用

    不是很推荐,这个索引巨迷惑,我没搞懂,似乎是替换法?

    个人建议还是edit吧...

    实例如下

    kubectl set <资源类型> <资源对象> <修改值>
[root@ansible ~]# kubectl set image deployment nginx nginx=nginx:v2
deployment.apps/nginx image updated
修改后如下
spec:
      containers:
      - image: nginx:v2 # 这里发生了变更
        imagePullPolicy: Always

    replace-破坏性更新

    该指令和apply功能类似,不同的是replace会先删除会产生冲突的资源(该yaml需要创建,但是已经存在的资源),然后在部署(破坏性更新),类似于删除并部署;而apply会尝试在现有资源上去修改,实在无法实现修改的才使用删除手段

    apply更温柔,对于改名字啊这种操作,直接就修改了(类似于你edit去修改),而replace则暴力一些,直接全删了

    ⚠️警告:这是破坏性更新,会导致不可控的后果,包括由于需要重新创建容器而导致资源请求数量激增等(特别是你的pod里面有链接mysql的,可能会一波干翻你的mysql)

    具体语法如下

    kubectl replace nginx

    查看日志

    在K8s中,POD是最小单位

    一个POD中可能包含一个到复数个容器

    因为上述的原因,因此我们查看日志需要分为两种情况,即为复数容器POD和单个容器POD

    前者查看较为复杂,后者则较为简单

    单个容器查看

    具体命令如下

    可选是否添加-f 添加的话代表及时追踪最新的日志(退出用crtl c),反之则显示从请求成功后的日志

    可选加入tail x只显示最后x行(不需要你用管道符了)

    kubectl logs <pod-name> -n <namespace>
# 实例
[root@ansible ansible]# kubectl logs pods/nginx-7854ff8877-9zq92  -n default


    # 添加tail
[root@ansible ansible]# kubectl logs -f pods/nginx-7854ff8877-9zq92  -n default --tail 3
2024/01/29 12:51:08 [notice] 1#1: start worker process 31
2024/01/29 12:51:08 [notice] 1#1: start worker process 32
2024/01/29 12:51:08 [notice] 1#1: start worker process 33

    多个容器查看

    多个容器查看要多一个步骤,即为指定你要查看该POD下的第几个容器

    命令如下

    使用关键字-c

    kubectl logs <pod-name> -c <container-name>
# 实例 查看nginx-7854 POD中的nginx容器
kubectl logs nginx-7854ff8877-9zq92 -c nginx

    关于如何查看POD内有哪些容器,可以使用kubectl get pod <pod-name> -o yaml去查询

    describe-获取资源对象详细信息

    这条命令是万精油命令,除非POD被你DELETE了(其实DELETE过程中你都还能用这个看),其他所有状态机下都可以用该命令去查看信息

    该命令可以查看各种资源的详细信息和该资源所绑定的其他信息

    例如你用describe去查看pod,会列出pod状态,label,关联svs,创建的容器信息等

    具体语法如下

    kubectl describe <对象类型> <对象名称> -n <命名空间>
# 实例
kubectl describe pod nginx-7854ff8877-9zq92 -n default

    top-查看资源负载

    使用该命令需要安装了metrics-server,该工具部署在集群之中,为集群提供了监控指标.

    该命令支持查询某个pod的负载和某个节点的负载

    命令如下

    kubectl top <node/pod> <node/pod-name> -n <namespace>
kubectl top node master-1.13x.cc -n default

    exec-执行命令

    exec和docker差不多,在容器中执行命令,支持交互式和执行式

    可以采用-it去引出shell,从而进入到容器中

    具体语法如下

    无需指定资源类型,因为只有pod支持该命令

    # 直接执行命令
kubectl exec <pod-name> -c <containerd> -- <命令>
kubectl exec nginx-7854ff8877-9zq92 -c nginx -- ls
# 采用交互式
kubectl exec  <pod-name>  -c <container-name> -it --  <shell>
kubectl exec  nginx-7854ff8877-9zq92  -c nginx -it --  /bin/sh
